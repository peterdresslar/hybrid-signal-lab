Battery generation is now recipe-driven.

Durable source pools live under `battery/data/sources/`. Canonical procedural
seed files use `*_seed.json` names so `build_battery.py` can auto-detect them.
Generated battery builds, manifests, and calibration artifacts should live in
their own versioned output directories under `battery/data/`.

Battery calibration remains a cloze-first process: the key calibration metric
is the probability and rank of the first target token.

## Recommended Workflow

### 1. Regenerate Procedural Seed Files

This refreshes all canonical `*_seed.json` files using recipe counts. It
overwrites existing files in place.

```bash
uv run -m battery.src.build_battery \
  --recipe battery/data/recipes/battery_4_recipe.json \
  --reseed
```

Notes:

- `code_comprehension` reseeding requires `OPENROUTER_KEY`.
- `domain_knowledge` is intentionally skipped by `--reseed`.
- `structural_copying` is generated inline during the build, not as a seed file.

### 2. Build Battery 4

```bash
uv run -m battery.src.build_battery \
  --outdir battery/data/battery_4 \
  --recipe battery/data/recipes/battery_4_recipe.json
```

If the canonical seed files exist under `battery/data/sources/`, the builder
will auto-load them. Otherwise it falls back to the older inline generators for
those types.

### 3. Run Calibration

Minimal calibration:

```bash
uv run -m battery.src.calibrate \
  --battery battery/data/battery_4/all_candidates.json \
  --model Qwen/Qwen3.5-2B-Base \
  --output battery/data/battery_4/calibration_qwen2b.jsonl
```

Heavier calibration with attention and hidden-state outputs:

```bash
uv run -m battery.src.calibrate \
  --battery battery/data/battery_4/all_candidates.json \
  --model Qwen/Qwen3.5-2B-Base \
  --output battery/data/battery_4/calibration_qwen2b.jsonl \
  --output-attentions \
  --output-hidden-states
```

### 4. Analyze Calibration Outputs

```bash
uv run -m battery.src.calibration_analyze \
  --battery-dir battery/data/battery_4
```

The analyzer now emits summaries by:

- `type`
- `tier`
- `family` when present
- `concept` when present
- `difficulty` when present

## References

- COUNTERFACT: [Meng et al., "Locating and Editing Factual Associations in GPT"](https://arxiv.org/abs/2202.05262), with the battery using the [`NeelNanda/counterfact-tracing`](https://huggingface.co/datasets/NeelNanda/counterfact-tracing) Hugging Face dataset for `factual_recall` and `factual_retrieval`.
- LAMBADA: [Paperno et al., "The LAMBADA dataset: Word prediction requiring a broad discourse context"](https://arxiv.org/abs/1606.06031), with the battery using the [`EleutherAI/lambada_openai`](https://huggingface.co/datasets/EleutherAI/lambada_openai) Hugging Face dataset for `cultural_memorized`.
- Targeted syntactic evaluation: [Marvin and Linzen, "Targeted Syntactic Evaluation of Language Models"](https://aclanthology.org/D18-1151/) and the accompanying [`BeckyMarvin/LM_syneval`](https://github.com/BeckyMarvin/LM_syneval) templates, which inform the agreement-style `syntactic_pattern` items.
- Wikipedia-derived domain knowledge items are curated from random article summaries and longer article-derived extracts using the Wikimedia APIs, including the [random summary endpoint](https://en.wikipedia.org/api/rest_v1/page/random/summary).

## Calibration Notes

- Calibration measures first-token target behavior, not full multi-token
  sequence likelihood.
- Prompt rendering is currently identity-only and recorded explicitly in the
  calibration artifacts via `prompt_render_version`.
- Calibration records now include target-token diagnostics such as:
  - `target_num_tokens`
  - `target_starts_with_space`
  - `target_first_token_str`
